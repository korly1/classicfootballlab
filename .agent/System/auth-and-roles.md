# Auth & Roles

## Access Levels

| Level  | Who             | How                            | What they can do        |
| ------ | --------------- | ------------------------------ | ----------------------- |
| coach  | Eduardo, Camila | Email/password via Supabase    | Full admin — all CRUD   |
| parent | Player families | Unique link + PIN (no account) | View their player's report |
| public | Anyone          | No auth                        | Landing page only       |

## Coach Auth

Standard Supabase email/password auth. After login, `src/proxy.ts` checks the user has a row in `public.coaches`. If yes, they pass. If not, they are redirected to `/login`.

```ts
// src/proxy.ts (Next.js 16 — proxy replaces middleware file name)
export async function proxy(req: NextRequest) {
  const supabase = createServerClient(...)
  const { data: { user } } = await supabase.auth.getUser()
  if (!user && req.nextUrl.pathname.startsWith('/admin')) {
    return NextResponse.redirect(new URL('/login', req.url))
  }
  const { data: coach } = await supabase
    .from('coaches').select('id').eq('id', user.id).maybeSingle()
  if (!coach) return NextResponse.redirect(new URL('/login', req.url))
  return NextResponse.next()
}

export const config = { matcher: ['/admin/:path*'] }
```

Adding a third coach in the future: create a Supabase auth user for them, insert a row in `public.coaches` with their auth user id. No code change needed.

## Parent Report Access (token + PIN)

No account required. The `share_token` lives on the player row and never changes (unless regenerated). The share link is: `https://lab.football/report/[token]`

The parent page always shows the player's most recent published evaluation.

Flow:

1. Parent opens the link
2. App shows a PIN entry screen
3. Parent enters the PIN Eduardo or Camila set
4. Server Action verifies PIN (bcrypt compare against `players.share_pin`)
5. On success, a session cookie is set scoped to that token
6. Report renders — always the latest `is_published = true` evaluation

PIN verification uses `adminClient` (service role) inside a Server Action. `adminClient` is never imported in Client Components.

## Setting Up the Share Link (Admin)

When a player is created, `share_token` is auto-generated by the database. `share_enabled` defaults to false and `share_pin` is null.

Eduardo or Camila activates sharing from the player profile:

1. Set a PIN (4-6 digits)
2. App bcrypt-hashes it and stores it in `players.share_pin`
3. Sets `players.share_enabled = true`
4. The share URL is displayed and ready to send

Sharing can be disabled at any time (`share_enabled = false`). The token can be regenerated to invalidate an old link.
